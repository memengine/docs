MemoryOS is a memory infrastructure layer for AI products. It gives you two connected systems: tenant-scoped memory for your own application, and a user-controlled Memory Passport layer for cross-agent sharing when a person wants multiple AI agents to remember them consistently.

MemoryOS exists so your team does not have to build extraction, retrieval, ranking, lifecycle cleanup, version history, permissions, consent UX, quota handling, and graceful degradation in-house just to make an AI product remember the right things about the right user at the right time.

## What problem does this solve?

Without MemoryOS, teams usually end up building three different systems at once:

- a memory pipeline that turns raw conversations into durable profile facts, goals, and preferences
- a retrieval layer that can rank the right memories quickly enough to use inside live AI calls
- a permissions and consent layer for deciding who is allowed to read or write which memories

MemoryOS gives you those pieces as one platform so your product team can focus on the user experience instead of stitching together memory extraction, vector search, context building, quota enforcement, lifecycle operations, and privacy controls from scratch.

## What MemoryEngine does

MemoryEngine is the runtime layer inside MemoryOS. It is not just a vector database wrapper.

On the write path, MemoryEngine:

- quality-gates noisy or unsafe ingestion attempts before they become jobs
- extracts durable memories from conversations using the extraction prompt
- resolves conflicts when new memories contradict older memories
- records version history for debugging and GDPR export
- stores memories in PostgreSQL and indexes searchable vectors separately

On the read path, MemoryEngine:

- retrieves candidate memories with hybrid ranking
- combines semantic relevance, importance, recency, and hot-tier cache results
- builds `system_prompt_addition` in `bullets`, `json`, or `xml`
- respects tenant isolation or Memory Passport category grants
- degrades safely when dependencies or quotas are unhealthy

On the operations path, MemoryEngine:

- decays stale memories over time
- auto-archives low-value inactive memories from search while keeping PostgreSQL history
- promotes frequently used high-value memories to the hot tier
- tracks extraction jobs, dead-letter failures, and dependency circuit states

## The two-layer identity model

For the standard tenant-scoped API, MemoryOS uses two identity layers on every integration:

1. **Tenant identity**
   Your MemoryOS API key identifies your company or workspace. All quotas, rate limits, webhooks, and billing controls are scoped at the tenant level.
2. **`external_user_id`**
   Inside your tenant, every end user is identified by an `external_user_id` that you choose, such as `customer-123`, `acct_7781`, or `user_42`.

That means the same `external_user_id` can exist in two different tenants without colliding, because MemoryOS always evaluates it inside tenant scope.

## The cross-agent layer

MemoryOS also supports a separate **Memory Passport** model for user-controlled sharing across agents.

- the user signs in to the consent app with email and OTP
- MemoryOS maintains a universal identity and internal `uui_...` token for agent-to-agent API calls
- an agent requests access to specific categories such as `preference` or `expertise`
- the user grants or revokes access through a consent flow
- the agent uses its own `agent_sk_...` key plus the user's UUI token to read or write only the categories that were approved

This lets multiple AI agents share memories about the same person without exposing the rest of that person's memory graph.

## What you send

For every write or retrieve request, you authenticate with a tenant API key and pass an `external_user_id` for the end user:

```http
Authorization: ApiKey mem_...
```

```json
{
  "external_user_id": "customer-123"
}
```
